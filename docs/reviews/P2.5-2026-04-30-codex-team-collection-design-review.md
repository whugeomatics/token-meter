## 检查结果

阶段边界：通过

- P2.5 明确只做 Codex team usage collection。
- P2.5 明确不做 Claude Code、Cursor、本地模型网关、多 provider 自动路由。
- P2.5 明确不破坏 P1/P2 `/api/report` contract。

身份归属：通过

- device token 被定义为可追溯到具体使用人的凭证。
- 服务端维护 `device_token -> team_id + user_id + device_id + status`。
- 最终写入 usage event 的 `team_id`、`user_id`、`device_id` 必须来自 token 绑定。
- 客户端自报 `client_user_id`、`client_device_id` 只用于一致性校验，不决定归属。

隐私：通过

- contract 禁止上传 prompt、response、raw JSONL、用户输入、模型输出和原始日志行。
- `event_key` 使用 `source_path_hash`，避免上传成员电脑完整本地路径。
- 错误响应和错误日志不得包含 token 明文或原始 JSONL。

幂等和重复统计：通过

- 服务端以 `team_id + user_id + device_id + event_key` 去重。
- collector retry 同一 batch 不应造成重复统计。
- 无 `token_count` 的 session 不生成 usage event，不做估算。

与 P2 兼容性：通过

- P2.5 复用 P2 delta 口径。
- P2.5 设计没有改变 P2 本地 ingestion 和 `/api/report`。
- 后续实现应优先复用 P2 parser/repository 边界，避免重复统计逻辑分叉。

团队展示：通过

- P2.5 已定义 `/api/team/report` 作为团队 dashboard 的只读 contract。
- dashboard 必须同时展示团队总览和每个用户的使用明细。
- 前端不得自行重新聚合原始 event，应消费后端 report contract。

## 实现前注意项

- 认证相关代码会进入 P2.5，实现前必须再次确认 token 明文不会进入日志或错误响应；如果产品要求后续复制 token，则 registry 明确按敏感数据处理。
- 首个实现版本可以使用手动预置 token 绑定，不需要组织管理后台。
- collector 本地 token 存储 MVP 可用用户 home 下配置文件，但要限制文件权限。
- 如果后续要支持多设备同一用户，`device_id` 必须稳定且可在 dashboard 中区分。

## 实现 Review 追加

状态：通过。

检查结果：

- 后端已拆 package，职责边界从单包拆为入口、HTTP、ingestion、domain、report、store、util。
- 前端已拆为 HTML、CSS、JS 三个静态资源文件。
- device token 鉴权查找使用 SHA-256 hash；为支持管理员后续复制，registry 同时保存可恢复 token。
- 服务端从 token binding 识别 `team_id`、`user_id`、`device_id`。
- 客户端自报 `client_user_id` / `client_device_id` 与 token binding 冲突时返回 `identity_conflict`。
- team usage event 使用 `team_id + user_id + device_id + event_key` 幂等去重。
- `/api/team/report` 返回团队总览、用户明细、设备明细、模型聚合、daily、user_daily。
- P1/P2 `/api/report` 未改 contract，P2 smoke test 通过。
- team token 绑定与高增长 usage event 已拆开存储：registry 保存 token hash、可恢复 token 和归属，月度分片保存事件。
- `--create-device-token` 可生成并绑定用户设备 token。
- 管理员页面/API 已受 `--admin-token` 或 `TOKEN_METER_ADMIN_TOKEN` 保护。
- 管理员可创建 teammate token；列表只返回 mask 后的 token，不返回完整 token 或 hash。
- 管理员可通过受 admin 鉴权保护的复制 API 获取完整 token，并可删除 token binding；token 不支持编辑。
- collector 模式已提前从入口分流，只使用 `--server-url` 上报，不依赖数据库，不启动 HTTP server，不初始化 server team registry。
- collector 支持 batch 上传，服务端限制单 batch 最多 500 条 event。
- 前端已拆成 Local 和 Team 两个独立 Tab，避免本机 sessions 与团队 users/devices 混在同一视图。

## Module 架构 Review 追加

状态：通过。

检查结果：

- P2.5 team 上报作为临时拆入需求，原始 design/tasks/acceptance 对 collector 轻量 jar 和三 module 边界记录不足；已补充 `P2.5-2026-05-08-module-architecture.md`。
- collector 不应依赖本地 SQLite checkpoint 或 retry queue DB；周期上报通过重新扫描最近窗口并依赖服务端 `event_key` 幂等去重保证不重复统计。
- collector jar 必须与 app jar 分离，不能包含 dashboard HTTP/API、report、store、admin、static assets 或 SQLite driver。
- 根 `AGENTS.md` 应保持索引化，P2.5 细节落入 P2.5 docs 和 archive，避免后续 agent 在当前阶段手册中读取过期细节。

环境记录：

- Codex mac 沙箱绑定本地 HTTP 端口失败于 `Operation not permitted` 时，P2.5 smoke test 会跳过 server bind 检查。
- 使用提升后的真实端口权限执行 `sh scripts/P2.5-2026-04-30-smoke-test.sh`，结果 `P2.5 smoke test passed`，真实 HTTP 链路、管理员页面/API、collector/server 分离已验证。
