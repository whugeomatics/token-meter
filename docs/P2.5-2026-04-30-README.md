# P2.5 2026-04-30 Codex Team Usage Collection

P2.5 的目标是在 P2 本地 Codex SQLite 采集能力基础上，增加团队视角的 Codex usage collection。

本阶段 MVP 只做 Codex。团队成员电脑部署轻量 collector，collector 只读本机 Codex sessions，在本地解析 JSONL，只向中央服务上传标准化 usage event。中央服务通过设备 token 识别团队、使用人和设备，并将事件写入中央 SQLite，供 dashboard 按 team、user、device、tool、model、date 聚合。

## MVP 边界

允许：

- 本地 collector 只读 `~/.codex/sessions`。
- collector 本地解析 Codex JSONL。
- collector 上传标准化 usage event。
- collector 不依赖本地数据库，周期运行时直接扫描本机 Codex sessions。
- collector 失败后可在下一次运行重新扫描最近窗口；重复事件由服务端幂等去重。
- collector 作为独立 module 和轻量 jar 分发，不包含 dashboard、SQLite、admin 或静态资源。
- 服务端通过设备 token 识别 `team_id`、`user_id`、`device_id`。
- 管理员通过受保护页面给 teammate 分配 device token。
- 服务端使用 `event_key` 幂等去重。
- 团队 usage event 按月拆分 SQLite，token 绑定单独保存在 registry SQLite。
- 中央 SQLite 按 team、user、device、tool、model、date 聚合。
- dashboard 以 Local 和 Team 两个 Tab 区分本机视角和团队视角。

禁止：

- 不上传 prompt 正文。
- 不上传 response 正文。
- 不上传 raw JSONL。
- 不接入 Claude Code。
- 不接入 Cursor。
- 不实现 P3 本地模型网关。
- 不实现复杂组织管理后台或 SSO。
- 不信任客户端自报 `user_id` 作为最终归属。

## 核心数据流

```text
member laptop Codex JSONL
  -> local collector
  -> shared core session parser
  -> normalized usage event batch
  -> server ingestion API
  -> device token binding lookup
  -> central SQLite monthly usage event shards
  -> /api/team/report
  -> team dashboard
```

## Module 架构

P2.5 采用三 module 架构：

- `token-meter-core`: 只放 app 和 collector 都需要的 session parser、team payload DTO、配置和通用工具。
- `token-meter-app`: 负责 dashboard/server、HTTP/API、admin、report、SQLite store、SQL resources 和静态资源。
- `token-meter-collector`: 负责 teammate 端轻量 collector CLI、Codex sessions 扫描和 `/api/team/ingest` 上报。

collector 不依赖 SQLite，不启动 HTTP server，也不包含 dashboard 静态资源或服务端 CRUD/API/report/admin 实现。完整边界见 `docs/milestones/P2.5-codex-team-collection/P2.5-2026-05-08-module-architecture.md`。

## 存储文件

默认 `--db` 指向目录时：

```text
token-meter-team-registry.sqlite
token-meter-team-YYYY-MM.sqlite
```

`token-meter-team-registry.sqlite` 保存 token hash、可恢复 token 和 `team_id/user_id/device_id` 绑定。可恢复 token 用于管理员后续复制给 teammate，因此 registry SQLite 应按敏感数据处理。`token-meter-team-YYYY-MM.sqlite` 保存按月增长的 usage event，便于长期运行后按月份清理历史文件。

## Dashboard 展示原则

P2.5 页面必须区分用户：

- 团队总览：团队总 token、input token、output token、session、活跃用户、活跃设备。
- 用户明细：每个用户的 token、session、设备数、最近上报时间。
- 设备明细：每台设备归属到哪个用户，以及该设备的用量。
- 模型聚合：团队维度和用户筛选后的模型用量。
- Team Models 明细：按 `date + user + model` 展示 token、session 数和 session 时间窗口，默认按日期倒序，可排序。
- 管理指标：展示 `usage_event_count`、`avg_tokens_per_session`、`avg_tokens_per_call`、`cache_hit_rate`、`reasoning_ratio`、`active_seconds`；`active_seconds` 按 session 内不超过 30 分钟的相邻 usage event 间隔累加，避免把跨天复用 session 的空闲时间算入；这些指标均由已入库 usage event 聚合得到，不需要上传 prompt/response。
- Team 筛选：页面可以按 `team_id` 切换不同团队。
- Team 页面使用内部 Tab 展示 Users、Models、Devices、Upload Health、Daily，避免数据增长后页面无限下滑。
- Upload Health 按 `team_id + user_id + device_id` 聚合，每个设备只展示最近 3 条上报和最后上报距现在的时间差；默认优先展示最久未上报的设备，不展示 token。
- 趋势：团队按天趋势，以及用户筛选后的按天趋势。

本地视角继续使用 P1/P2 `/api/report`。Local 页面也使用内部 Tab 展示 Sessions、Models、Daily，默认只展示最近 50 条 sessions，避免本地历史数据增长后页面过长。团队视角使用 P2.5 `/api/team/report`。

## 身份归属原则

每个成员设备配置一个上报 token。服务端维护 token 绑定关系：

```text
device_token -> team_id + user_id + device_id + status
```

上传 payload 里的 `user_id`、`team_id`、`device_id` 不可信。服务端必须以 token 绑定关系为准。客户端字段如果保留，只能用于一致性校验和问题排查；不一致时服务端应拒绝请求或记录授权错误，不能把事件归到客户端自报的人名下。

服务端提供 `--create-device-token` 生成并绑定 token。为支持管理员后续复制，registry 保存 token hash 和可恢复 token；普通 dashboard、错误响应和日志不得输出 token 明文。

## 管理员页面

P2.5 提供管理员页面分配 teammate token：

```text
/admin-login.html
/admin.html
```

启用方式：

```text
TOKEN_METER_ADMIN_TOKEN=<admin-token>
```

或：

```text
--admin-token=<admin-token>
```

未配置 admin token 时管理入口关闭。未登录管理员访问 `/admin.html` 或 `/api/admin/*` 会被拒绝。管理员创建 device token 后，binding 列表只展示 mask 后的 token，并提供复制按钮；完整 token 只通过受 admin 鉴权保护的复制 API 返回，不允许编辑 token，可以删除 token binding。

## 上报策略

collector 支持批量上报：

- 默认单 batch 最多 500 条 event。
- 可用 `--batch-size=<n>` 调小 batch。
- 重复上报依赖服务端 `event_key` 幂等去重。
- 周期性运行 collector 时可持续补报新增 event。
- Codex 关闭期间无需常驻；下次运行 collector 时批量补报。
- collector 成功或失败输出中可以包含 `upload_time`、`client_user_id`、`client_device_id`、`server_url`、日期窗口等非敏感诊断信息，不得包含 device token。

## 隐私原则

P2.5 只上传 usage delta 和必要归属字段。

Codex “提问数”不应直接按 `response_item.role=user` 统计，因为该事件可能包含 AGENTS、环境上下文和审批转录等系统注入内容；如后续实现，应优先评估 `event_msg.payload.type=task_started` 作为用户触发任务数的保守口径，并只上传计数不上传内容。

不得上传：

- prompt。
- response。
- raw JSONL 行。
- tool message 内容。
- 用户输入内容。
- 模型输出内容。

错误日志也不得包含上述内容。
