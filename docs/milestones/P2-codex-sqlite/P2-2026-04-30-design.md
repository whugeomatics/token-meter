# P2 Codex SQLite Design

## 当前阶段

P2：SQLite 持久化与增量采集。

P1 已完成只读实时扫描和 dashboard。P2 的目标是把 Codex usage delta 持久化到本地 SQLite，避免每次请求都全量扫描 JSONL，并为后续多工具统计打基础。

## 范围

P2 只支持 Codex。

允许：

- 新增本地 SQLite 文件。
- 新增 ingestion CLI。
- 从 SQLite 查询 `/api/report`。
- 保持 P1 dashboard 和 API contract 兼容。

禁止：

- 不实现 Claude Code。
- 不实现 Cursor。
- 不实现本地模型网关。
- 不写 `.codex`。
- 不存 prompt/response 正文。
- 不做云同步、登录、费用估算。

## 架构

P2 后端拆成四个职责：

1. JSONL reader：只读 Codex session 文件。
2. Usage normalizer：沿用 P1 规则，把 cumulative snapshot 转成 delta event。
3. SQLite repository：保存 source checkpoint 和 usage delta。
4. Report service：从 SQLite 聚合 P1 `/api/report` 响应。

P2 不需要前端大改。dashboard 继续消费 `/api/report`。

2026-04-30 mac 修复后，dashboard 服务启动前先执行一次本地 Codex ingestion，避免页面只读取空 SQLite；页面 API 仍保持只读。

## 数据流

```text
Codex JSONL
  -> ingestion CLI / dashboard startup ingestion
  -> total_token_usage delta
  -> SQLite usage_events
  -> /api/report
  -> dashboard
```

## SQLite 文件

默认从 2026-04-30 mac 修复起改为月分片目录：

```text
%USERPROFILE%\.token-meter\sqlite\token-meter-YYYY-MM.sqlite
```

可通过 `TOKEN_METER_DB` 覆盖。覆盖值指向 `.sqlite` / `.db` 文件时，继续使用单 SQLite 文件兼容模式；覆盖值指向目录时，按月分片。

选择理由：

- 不写 `.codex`。
- 和 dashboard 工具自身数据放在独立目录。
- 月分片便于后续备份或删除历史数据，避免长期只写一个大 SQLite 文件。

## 增量采集策略

每个 source file 记录：

- path。
- size。
- modified time。
- last processed line。
- fingerprint。
- status。

常规追加场景只重新处理发生变化的文件；未变化文件直接跳过。

对于 changed source file，P2 从头重放该文件，并通过 `usage_events.event_key` 去重，避免重复统计。

这样做的原因是 token delta 需要上一条 cumulative snapshot。当前 schema 不持久化上一条 snapshot，如果只从 `last_line + 1` 读取，第一条新增 snapshot 会缺少前置基线，导致 delta 被错算为全量值。

## 统计口径

P2 必须保持 P1 口径：

- 使用 `payload.info.total_token_usage` 相邻累计快照 delta。
- 跳过相同累计快照。
- 不直接累加 `last_token_usage`。
- token 字段入库时必须是 delta，不是 cumulative。

如果 Codex session 文件没有 `event_msg.payload.type=token_count`：

- P2 只记录 source file checkpoint。
- 不生成 usage event。
- 不把该 session 展示为有 token 消耗的 session。
- 不根据消息正文或文件内容估算 token。

## API 兼容

P2 继续提供：

```text
GET /api/report
```

查询参数和返回字段保持 P1 contract 不变。

P2 内部可从 SQLite 聚合：

- summary。
- daily。
- models。
- sessions。

## 错误处理

解析单行失败：

- 记录错误摘要。
- 不记录原始 JSONL。
- 不影响其他文件。

数据库失败：

- ingestion 返回非 0。
- 不推进失败行之后的 checkpoint。

日志目录不存在：

- ingestion 输出 0 events。
- `/api/report` 返回空数据。

## 验收策略

P2 验收必须证明：

- 首次导入能写入 usage event。
- 二次导入不会重复写入。
- 新增日志后只导入新增部分。
- `/api/report` 与 P1 contract 兼容。
- 不写 `.codex`。
- 不存 prompt/response 正文。

## 后续阶段

P2 不做 provider 抽象，但 schema 中保留 `tool` 字段，P2 固定为 `codex`。

多 provider、Claude Code、Cursor 和本地网关都留到后续阶段。
