# P2.5 Codex Team Usage Collection Design

## 当前阶段

P2.5：Codex 团队用量采集。

P2 已完成本地 SQLite 持久化和增量采集。P2.5 的目标是在不改变 P2 本地统计能力的前提下，增加团队成员电脑到中央服务的主动上报链路。

## 范围

P2.5 只支持 Codex。

允许：

- 新增本地 collector 模式。
- 新增服务端 team ingestion API。
- 新增设备 token 绑定。
- 新增中央 usage event 存储字段。
- 新增团队视角聚合和 dashboard 展示。
- 新增团队 report API，支持团队总览和每个用户的使用明细。

禁止：

- 不实现 Claude Code。
- 不实现 Cursor。
- 不实现本地模型网关。
- 不上传 prompt、response、raw JSONL。
- 不实现复杂账号系统或 SSO。
- 不允许客户端决定事件最终归属。

## 架构

P2.5 增加两个角色：

1. Collector：运行在团队成员电脑上，只读本机 Codex sessions，本地解析并上传标准化 usage event。
2. Server：接收 event batch，通过设备 token 识别归属，写入中央 SQLite，并提供团队聚合查询。

```text
Codex JSONL on member laptop
  -> collector local parser
  -> normalized usage event batch
  -> POST /api/team/ingest
  -> device token binding lookup
  -> central SQLite
  -> team dashboard
```

代码按三个 Maven module 拆分：

- `token-meter-core`: 只放 app 和 collector 都需要的 Codex session parser、team payload DTO、配置和通用工具。
- `token-meter-app`: 负责 server/dashboard、HTTP/API、admin、report、SQLite CRUD/store、SQL resources 和静态资源。
- `token-meter-collector`: 负责 teammate 端轻量 collector CLI 和上报逻辑。

collector jar 必须保持轻量，不包含 app jar、dashboard 静态资源、SQLite driver、server API/report/store/admin 实现。完整 module 边界见 `P2.5-2026-05-08-module-architecture.md`。

## Collector 设计

collector 复用 P2 Codex JSONL 解析和 delta 生成规则。

职责：

- 扫描 `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`。
- 只读取 `token_count` / `total_token_usage` 相关字段。
- 把 cumulative snapshot 转换成 delta event。
- 生成稳定 `event_key`。
- 不记录或上传 prompt、response、raw JSONL。

collector 不保存本地 checkpoint，也不依赖 SQLite。周期性上报直接扫描最近窗口内的 Codex sessions，重复上报由服务端按 `team_id + user_id + device_id + event_key` 幂等去重。

## Server 设计

server 新增 `POST /api/team/ingest`。

职责：

- 校验 bearer device token。
- 从 token 绑定得到 `team_id`、`user_id`、`device_id`。
- 校验 payload。
- 对 `event_key` 幂等去重。
- 写入中央 SQLite。
- 更新 token `last_seen_at`。
- 返回 accepted/duplicate/rejected 统计。

server 不信任客户端自报身份。payload 中的 `client_user_id`、`client_device_id` 只作为一致性校验字段。

## 设备 token 和使用人

P2.5 必须能回答“这个 token 是谁的 token”。

服务端维护：

```text
device_token -> team_id + user_id + device_id + status
```

最终写入 usage event 的归属字段必须来自 token 绑定，而不是来自客户端 payload。

如果客户端自报 `client_user_id` 与 token 绑定不一致，服务端拒绝该 batch，避免错误归属或冒名上报。

## 管理员分配 token

P2.5 需要一个管理员页面，用于给 teammate 分配 collector 上报 token。

MVP 不引入完整账号系统，只使用服务启动时配置的 admin token 保护管理入口：

```text
TOKEN_METER_ADMIN_TOKEN=<admin-token>
```

或：

```text
--admin-token=<admin-token>
```

管理页面能力：

- 管理员登录。
- 查看已分配的 token 绑定列表，列表只展示 mask 后的 token。
- 输入 `team_id`、`user_id`、`device_id`、`device_name` 创建 teammate token。
- 复制完整 device token；完整 token 只通过受 admin 鉴权保护的复制 API 返回。
- 删除 token binding，删除后该 token 不能继续上报。
- 不允许编辑 token；绑定错了应删除后重新创建。

安全约束：

- 未配置 admin token 时管理入口关闭。
- 未登录管理员不能访问 `/admin.html` 和 `/api/admin/*`。
- 管理列表不直接展示完整 device token 或 hash，只展示 `token_preview`。
- 为支持后续复制，registry 保存可恢复 token；registry SQLite 应按敏感数据处理。
- admin token 不写入日志、DB、HTML、JS。

## 数据库方向

P2.5 应在 P2 SQLite schema 基础上扩展，而不是重写整个存储层。

团队中央库必须拆成两类文件：

```text
token-meter-team-registry.sqlite
token-meter-team-YYYY-MM.sqlite
```

`token-meter-team-registry.sqlite` 保存低增长的 token 绑定；`token-meter-team-YYYY-MM.sqlite` 保存高增长的 team usage event。这样长期运行后可以按月份清理历史 usage event 文件，而不会误删 token 绑定。

中央 usage event 至少需要支持：

- `team_id`
- `user_id`
- `device_id`
- `tool`
- `session_id`
- `model`
- `timestamp`
- `input_tokens`
- `output_tokens`
- `total_tokens`
- `event_key`
- `received_at`

P2 的本地 `/api/report` contract 保持兼容。P2.5 团队视角新增 endpoint：

```text
GET /api/team/report
```

该 endpoint 的 contract 见 `docs/contracts/P2.5-2026-04-30-team-report-api.md`。

## Dashboard 设计

P2.5 dashboard 最小新增团队视角：

- 团队总览：team 总 token、input token、output token、session、活跃用户、活跃设备。
- 用户明细：每个 user 的 token、session、设备数、最近上报时间。
- 设备明细：每个 device 的 token、session、归属 user。
- 模型聚合：团队总模型用量，以及 user 筛选后的模型用量。
- 趋势：团队按天趋势，以及 user 筛选后的按天趋势。

页面上必须能看到：

- 所有用户合计的总用量。
- 每个用户各自的使用情况。
- 某个用户展开或筛选后的设备、模型、日期明细。

本地视角和 P1/P2 `/api/report` 不应被破坏。

页面必须有两个明确 Tab：

- `Local`: 只展示本机 `/api/report`。
- `Team`: 只展示 `/api/team/report`。

两个 Tab 不能把 Local sessions 和 Team users/devices 混在同一视图里。

## 上报设计

collector 不使用本地 P2 SQLite checkpoint，也不写本地 collector DB。周期性上报时，collector 直接扫描最近窗口内的 Codex sessions，生成标准化 usage event 后调用服务端 ingestion API。

上报策略：

- 单 batch 最多 500 条。
- 周期性运行时上传最近窗口内 event。
- Codex 关闭期间无需后台运行；下次 collector 启动时批量补报。
- 重复上报依赖服务端 event key 去重。
- 网络失败或 5xx 后，下次运行重新扫描最近窗口并重试；400、401、403、409 不进入无限重试。
- 任何日志或错误不得输出 token 明文。

## 隐私

P2.5 的隐私边界比 P2 更重要，因为数据跨设备上报到中央服务。

不得上传：

- prompt。
- response。
- raw JSONL。
- 用户输入内容。
- 模型输出内容。
- 原始日志行。
- 本机完整文件路径。

路径相关字段如果用于 `event_key`，必须先在 collector 本地 hash 后再上传。

## 验收策略

P2.5 验收必须证明：

- collector 可从 fixture 生成标准 usage event。
- 上传 payload 不含敏感正文。
- 未知 token 被拒绝。
- token 能绑定到明确使用人。
- 客户端自报 user 与 token 绑定冲突时被拒绝。
- 重复上传不重复统计。
- 断网或 5xx 后 collector 可重试。
- 团队 report 能按 user/device/model/date 聚合。
- dashboard 能同时展示团队总用量和每个用户的使用情况。
