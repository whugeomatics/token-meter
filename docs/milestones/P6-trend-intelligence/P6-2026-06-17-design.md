# P6 Team Trend Intelligence Design

## Product Direction

P6 keeps Token Meter focused on team token statistics. It should make the dashboard answer: "What changed, and why?"

The design deliberately avoids request proxying, provider routing, gateway behavior, pricing, and billing. Those are outside the current product direction.

## Data Flow

```text
canonical usage events + upload health
        |
        v
existing period comparison windows
        |
        v
trend insight aggregation
        |
        v
Team dashboard What changed + alerts + Markdown summary
```

P6 should derive all insights from existing stored facts. It should not add a new collector payload unless an existing field is demonstrably insufficient for the trend contract.

## Backend Components

- Report layer computes `trend_insights` from current and previous period aggregates.
- Driver aggregation groups by user, device, model, and tool.
- Signal detection applies simple deterministic thresholds first.
- Upload health trends reuse accepted, duplicate, rejected, latest upload, and device freshness data.

Thresholds should start conservative and local to the report layer. Avoid adding a rule engine or user-configurable alert system in P6.

## Frontend Components

- Add a `What changed?` section to Team overview.
- Show top positive and negative drivers by user, device, model, and tool.
- Show compact trend signals near the existing top summary so report trust issues are visible.
- Add a copy Markdown summary action for the current filters and period.
- Improve empty-state text by checking whether other filters have data.

P6 UI should support scanning and repeated use. Avoid marketing-style explanation blocks.

## Filtering

Filtering remains a support feature, not the center of P6.

Required filters:

- team;
- tool;
- model;
- user;
- device when useful for Upload Health.

The current filters should be visible, and there should be one clear reset action.

## Privacy

P6 must preserve the P5 privacy boundary. Trend insight generation must not read, store, return, log, or export prompt/response content, raw log lines, local full source paths, admin tokens, device tokens, or token hashes.

## Testing

Backend tests should cover:

- driver delta calculation;
- zero previous-period behavior;
- sorting by absolute delta;
- signal thresholds;
- upload health trend inputs;
- privacy boundary fields not appearing in trend responses.

Frontend tests should cover:

- Team trend section rendering;
- empty-state guidance with and without data in other filters;
- Markdown summary content for current filters;
- Local dashboard remains unaffected.
