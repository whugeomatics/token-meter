# P5 2026-05-24 Unified CLI Usage Metrics Design

## 目标

P5 在 P4 已经支持 Codex 和 Claude Code local/team 统计的基础上，统一 CLI usage metrics。完成后，Codex 和 Claude Code 的 event 字段、fallback 规则和 report 派生指标含义一致；未来 CLI 工具可以通过新增 source mapping 进入同一套统计模型。

P5 允许少量代码调整，但只做让现有实现贴合 unified metrics contract 的最小修改。

## 背景

P4 已经实现：

- Local 和 Team 默认同时采集 Codex 与 Claude Code。
- `tool=codex|claude-code` 筛选。
- `tools` 聚合和 tool ranking。
- period comparison 下的 tool delta。
- Claude Code cache token 归一化和本地 JSONL 去重。

P5 要解决的问题不是再接一个工具，而是避免 Codex 和 Claude Code 在同名字段、cache rate、net tokens 或缺失字段处理上产生不同语义。

## 设计原则

字段分为两类：

- 事实字段：来源工具明确上报，或 collector 可以在不读取正文的情况下稳定推导。事实字段可以进入 event、payload 和 DB。
- 派生指标：由 report 层根据事实字段计算。派生指标不作为新的历史事实写入 DB。

这个边界让未来 CLI 可以补充自己的 source mapping，同时避免历史数据绑定到某个临时计算公式。

## Canonical Event

P5 canonical usage event 事实字段：

```text
event_key
tool
provider
source_kind
source_quality
session_id
model
timestamp
input_tokens
cached_input_tokens
output_tokens
reasoning_output_tokens
total_tokens
source_event_key
client_user_id
client_device_id
```

`provider` 是可选字段。P5 可以在 schema 中预留，但不要求 dashboard 增加 provider filter 或 provider ranking。

## Report 派生指标

Local `/api/report` 和 Team `/api/team/report` 必须使用同一公式：

```text
net_input_tokens = max(input_tokens - cached_input_tokens, 0)
net_total_tokens = net_input_tokens + output_tokens + reasoning_output_tokens
cache_rate = cached_input_tokens / input_tokens
```

`input_tokens` 为 0 时，`cache_rate` 为 0。`cached_input_tokens` 大于 `input_tokens` 时，展示层可以 clamp 到 1.0，但 net input 不得为负数。

summary、daily、models、sessions、tools 和 comparison 都应复用同一计算路径或同一 helper，避免 Local/Team 口径分叉。

## Source Mapping

Codex 和 Claude Code 各自保留来源解析逻辑，但都输出 canonical event 字段。

Codex:

- `tool=codex`。
- `source_kind=local_jsonl`。
- token delta 来自 Codex usage metadata。
- cache token 缺失时为 0。
- 不读取或上传 prompt、response、raw JSONL。

Claude Code:

- `tool=claude-code`。
- `source_kind=local_jsonl|otel_metric|otel_log|hook_metadata|fixture`。
- `input_tokens` 使用 Claude `input_tokens + cache_creation_input_tokens + cache_read_input_tokens`，保证 canonical input 是 gross input。
- `cached_input_tokens=cache_creation_input_tokens + cache_read_input_tokens`。
- 优先使用 `message.id` hash 去重。
- 不读取或上传 prompt、response、raw API body、raw transcript 或 tool input/output。

## 代码调整边界

P5 代码调整应从检查不一致开始，只修改 contract 要求的最小范围：

- 如果 event/payload 已有 canonical 事实字段，保持兼容，不重命名。
- 如果缺少 `source_kind`、`source_quality` 或 `reasoning_output_tokens` 等事实字段，评估是否需要补齐到 DTO、DB 和 report。
- 如果 Local 和 Team 派生指标公式不同，收敛到同一 helper 或同一实现。
- 如果 dashboard 同名指标显示不同语义，修正展示和文案。
- 不新增新工具 adapter。
- 不新增 provider 聚合 UI。

## 数据流

```text
Codex or Claude Code source metadata
  -> source-specific parser
  -> canonical usage event facts
  -> local store or team ingest
  -> report aggregation
  -> derived metrics
  -> Local/Team dashboard
```

## 隐私和安全

P5 不放宽任何 P4 隐私边界。

禁止进入 event、payload、DB、report、stdout 和日志：

- prompt。
- response。
- raw JSONL。
- raw API body。
- raw transcript。
- user input。
- model output。
- tool input payload。
- tool output payload。
- admin token。
- device token 明文。
- device token hash。
- 完整本地路径。

修改认证、admin token、device token 或隐私相关代码前，必须再次说明安全影响。

## 非目标

- 不接入 Cursor。
- 不接入 Gemini CLI。
- 不实现本地模型网关。
- 不实现 provider 自动路由。
- 不做费用估算。
- 不做预算或告警。
- 不重构三 module 架构。

## 验证策略

- 单元测试覆盖 canonical fallback 和派生指标公式。
- Local report fixture 验证 Codex 和 Claude Code 字段语义一致。
- Team report fixture 验证 tool、model、session、daily 和 comparison 聚合一致。
- 隐私 fixture 验证禁止字段不进入 payload、DB、report、stdout 或日志。
- JS 语法检查验证 dashboard 改动。
- Maven test/package 验证 Java 改动。
