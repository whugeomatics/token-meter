# P6 Incremental Team Collection Design

## Product Direction

P6 keeps Token Meter focused on reliable team token statistics. Before adding richer trend UI, teammate collection must stop behaving like full-window re-upload.

The design deliberately avoids request proxying, provider routing, gateway behavior, pricing, and billing. Those are outside the current product direction.

## Current Behavior And Problem

The collector already keeps a local SQLite state table keyed by `event_key`. This prevents local duplicate storage.

However, upload currently reloads all stored events in the configured lookback window and posts them again. The server deduplicates by `event_key`, so report totals remain correct, but repeated runs create duplicate upload counts and make Upload Health noisy.

P6 changes upload selection from "all events in lookback window" to "events after the last successful upload cursor".

Local dashboard ingestion is a separate path. Codex local ingestion already has source file checkpointing and event-key idempotency. Claude Code local ingestion is currently idempotent by event key, but its source scan is less checkpoint-driven than Codex. P6 may tighten Claude Code local checkpointing as a small reliability cleanup, but the main product fix is Team collector upload cursor behavior.

## Data Flow

```text
local Codex / Claude Code usage metadata
        |
        v
collector scan
        |
        v
collector local state
        |
        v
pending events after upload cursor
        |
        v
/api/team/ingest
        |
        v
advance cursor only after successful upload
```

The scan can still inspect the local log source to discover new events. The incremental guarantee is about upload selection, not about never opening older files.

## Collector State

Add cursor state to the collector-side SQLite database.

Required fields:

- cursor name or scope, initially one cursor for the teammate device upload stream;
- last successful event timestamp;
- last successful event key;
- updated_at.

The timestamp plus event key pair avoids skipping events when multiple events share the same timestamp.

## Upload Selection

At collector run start:

1. Capture a fixed cutoff time.
2. Scan Codex and Claude Code usage metadata from the upload cursor lower bound when a cursor exists; use the configured initial lookback window only for the first run with no cursor.
3. Insert discovered events into local collector state with `INSERT OR IGNORE`.
4. Select pending events where `(event_timestamp, event_key)` is after the cursor and `event_timestamp <= cutoff`.
5. Upload selected events in deterministic order.
6. Advance cursor to the last selected event only if the full upload succeeds.

When there are no pending events, send an empty payload heartbeat and do not advance the cursor.

This distinction matters for teammate downtime. If a collector is not run for longer than the default lookback window, it must still discover and upload events after the last successful cursor instead of silently skipping them.

## Failure Compensation

P6 uses a conservative compensation model:

- If upload fails, do not advance the cursor.
- The next run retries from the old cursor.
- Some already-accepted events may be resent after a partial failure; server-side `event_key` deduplication handles that.
- P6 does not add a separate failed-batch queue unless later evidence shows cursor-level retry is insufficient.

This keeps the implementation small while preserving correctness.

## Dashboard Impact

No large new dashboard surface is required for P6.

Upload Health should improve indirectly because normal repeated runs without new events no longer inflate duplicate counts. If useful, the UI may later show collector cursor or collector version, but that is not required for P6 implementation.

## Privacy

P6 must preserve the P5 privacy boundary. Cursor state must store usage metadata only and must not store prompt/response content, raw log lines, local full source paths, admin tokens, device tokens, or token hashes.

## Testing

Collector tests should cover:

- first run uploads events and advances cursor;
- second run with no new events uploads no usage events and keeps duplicate count at zero;
- new events after cursor are uploaded on the next run;
- same-timestamp events use event key tie-breaker and are not skipped;
- collector downtime longer than the default lookback window does not skip events after the cursor;
- failed upload does not advance cursor;
- empty successful run still posts heartbeat;
- existing Local report, Team report, and P5 token formulas remain unchanged.
