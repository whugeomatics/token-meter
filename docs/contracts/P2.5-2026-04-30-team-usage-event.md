# P2.5 Team Usage Event Contract

## 目标

定义 collector 上传给中央服务的标准化 Codex usage event。

P2.5 复用 P2 统计口径：Codex `total_token_usage` 相邻累计快照转成非负 delta。上传的是 delta event，不是 cumulative snapshot。

## Event 字段

单条 event：

```json
{
  "event_key": "codex|session|source|line|tokens",
  "tool": "codex",
  "session_id": "session-id",
  "model": "gpt-5.1-codex",
  "timestamp": "2026-04-30T12:34:56Z",
  "input_tokens": 100,
  "output_tokens": 20,
  "total_tokens": 120,
  "source_event_key": "optional-source-event-key",
  "client_user_id": "optional-user-id",
  "client_device_id": "optional-device-id"
}
```

字段规则：

- `event_key`: 必填。全局幂等键，服务端对同一 `team_id + user_id + device_id + event_key` 去重。
- `tool`: 必填。P2.5 固定为 `codex`。
- `session_id`: 必填。Codex session 标识。
- `model`: 必填。模型名称，缺失时使用 `unknown`。
- `timestamp`: 必填。事件时间，ISO-8601 UTC。
- `input_tokens`: 必填，非负整数。
- `output_tokens`: 必填，非负整数。
- `total_tokens`: 必填，非负整数，必须等于或大于 `input_tokens + output_tokens` 的可解释总量。
- `source_event_key`: 可选。用于保留 collector 本地来源事件键。
- `client_user_id`: 可选，只用于服务端一致性校验，不决定归属。
- `client_device_id`: 可选，只用于服务端一致性校验，不决定归属。

## 不允许字段

event 不得包含：

- prompt。
- response。
- raw JSONL。
- message content。
- user input。
- model output。

如果 collector 无法从 Codex session 中读取 `token_count` / `total_token_usage`，不得伪造 event，也不得根据正文或文件大小估算 token。

## Event Key

`event_key` 必须稳定。MVP 沿用 P2 规则：

```text
codex|<session_id>|<source_path_hash>|<line_number>|<total_tokens>|<input_tokens>|<output_tokens>
```

说明：

- `source_path_hash` 使用本地路径哈希，不上传成员电脑的完整本地路径。
- `line_number` 来自 JSONL 行号。
- token 字段来自 delta event。

## 服务端写入规则

服务端写入中央库时应补充：

```text
team_id
user_id
device_id
received_at
```

这些字段来自服务端 token 绑定和服务端接收时间，不信任客户端自报。

## 服务端存储分片

团队 usage event 是长期增长数据，必须按事件本地日期所属月份拆分 SQLite：

```text
token-meter-team-YYYY-MM.sqlite
```

例如 `2026-04-30` 的事件写入：

```text
token-meter-team-2026-04.sqlite
```

设备 token 绑定不放入月度事件分片，统一保存在 `token-meter-team-registry.sqlite`。

## 聚合维度

中央库至少支持以下聚合：

- date。
- team。
- user。
- device。
- tool。
- model。
- session。

P2.5 只要求 `tool=codex`。
