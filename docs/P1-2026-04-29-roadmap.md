# Roadmap

## 总目标

做一款本地模型使用统计与路由工具，最终支持 Codex、Claude Code、Cursor、命令行脚本和自研工具统一接入，统计每次会话、每个模型、每天、每周、每月的 token 消耗和使用时长。

长期形态不是简单 dashboard，而是一个本地模型网关：

```text
Coding tools / IDE / CLI
        |
        v
Local Model Gateway
        |
        v
OpenAI / Anthropic / Gemini / DeepSeek / local models
        |
        v
Usage database + Dashboard
```

## 阶段 0：可行性调研

目标：确认每个工具能以什么方式接入统计链路。

需要做的事：

- 调研 Codex 本地日志字段。
- 调研 Codex 是否支持自定义 OpenAI-compatible base URL。
- 调研 Claude Code 是否支持自定义 Anthropic-compatible base URL。
- 调研 Cursor 是否能通过 API key、base URL、代理或日志方式统计。
- 判断每个工具适合使用“日志读取”还是“本地网关代理”。

产出物：

- `docs/research/codex.md`
- `docs/research/claude-code.md`
- `docs/research/cursor.md`
- `docs/research/provider-compatibility.md`

验收标准：

- 每个工具至少有一个可行接入路径。
- 明确哪些字段是真实统计，哪些字段只能估算。
- 明确哪些能力需要后续实测。

## 阶段 1：Codex Dashboard MVP

目标：先聚焦 Codex，做一个只读 dashboard。

范围：

- 读取本机 Codex session JSONL 日志。
- 展示今日、7 天、30 天、本月统计。
- 展示总 token、input、cached input、output、reasoning output、net usage。
- 展示按天、按模型、按会话的排行。
- 展示会话活动时长估算。

不做：

- 不代理 Codex 请求。
- 不修改 Codex 配置。
- 不存储 prompt 内容。
- 不做费用精确结算。
- 不适配 Claude Code、Cursor。

产出物：

- `docs/P1-2026-04-29-mvp-codex-dashboard.md`
- `docs/acceptance/P1-2026-04-29-mvp-codex-dashboard.md`
- 本地可启动 dashboard。

验收标准：

- 能在本机启动 dashboard。
- dashboard 能读取真实 Codex 日志。
- API 和页面展示数据一致。
- 统计口径写入文档。

## 阶段 2：本地持久化与增量采集

目标：从“每次页面请求实时扫描日志”升级到“增量采集 + SQLite”。

需要做的事：

- 设计 normalized usage event schema。
- 建 SQLite 表：sessions、usage_events、models、providers。
- 实现日志增量导入。
- 记录已处理文件和 offset，避免重复统计。
- 增加导入命令和重新扫描命令。

产出物：

- `docs/contracts/usage-event.md`
- `docs/contracts/database-schema.md`
- `docs/acceptance/sqlite-ingestion.md`

验收标准：

- 重复导入不会重复计数。
- 可以按天、周、月、模型、会话查询。
- 数据库可删除后重新构建。

## 阶段 3：Codex 团队用量采集

目标：在 P2 本地 Codex SQLite 采集能力基础上，增加团队成员电脑到中央服务的主动上报链路。

需要做的事：

- 拆分 `token-meter-core`、`token-meter-app`、`token-meter-collector` 三个 module。
- 实现 teammate 端轻量 collector，只读取本机 Codex sessions 并上报 usage event。
- 服务端通过 device token 绑定 team、user、device。
- 服务端按月保存 team usage event，并保留 token registry。
- Dashboard 增加 Local 和 Team 两个视角。
- Local 和 Team 支持 Day、Week、Month 自然周期对比。
- 不上传 prompt、response、raw JSONL、device token 明文或 token hash。

产出物：

- `docs/P3-2026-04-30-README.md`
- `docs/contracts/P3-2026-04-30-team-ingestion-api.md`
- `docs/contracts/P3-2026-04-30-team-report-api.md`
- `docs/contracts/P3-2026-04-30-team-usage-event.md`
- `docs/contracts/P3-2026-04-30-device-token.md`
- `docs/milestones/P3-codex-team-collection/P3-2026-04-30-design.md`
- `docs/milestones/P3-codex-team-collection/P3-2026-05-08-module-architecture.md`
- `docs/acceptance/P3-2026-04-30-codex-team-collection.md`

验收标准：

- Collector jar 不包含 dashboard、SQLite、admin 或静态资源。
- 重复上报不会重复统计。
- Team report 可按 team、user、device、model、date 聚合。
- Local/Team period comparison 与文档口径一致。
- P1/P2 `/api/report` 兼容查询不被破坏。


## 阶段 4：Claude Code 本地与团队用量采集

目标：在 P3 Codex team collection 已跑通的基础上，让 Local dashboard 和 teammate collector 都能采集 Claude Code usage metadata，使本地统计和团队统计从单一 Codex 扩展到至少两个真实 AI CLI 工具。

需要做的事：

- 调研并记录 Claude Code 官方 usage 上报入口，例如 OpenTelemetry metrics、tool activity、cost/usage metadata、hooks 可提供的 transcript path 等。
- 设计 Claude Code usage event contract，并明确哪些字段来自官方 usage metrics，哪些字段只能缺省或估算。
- 扩展 local ingestion，使 dashboard 启动和 `/api/ingest` 在有本地数据时同时采集 Codex 与 Claude Code。
- 扩展 collector，使其默认一次运行同时采集 Codex 和 Claude Code，并按 `tool=claude-code` 上报 usage metadata。
- 保留 `--collect-claude-code` 作为旧脚本兼容入口，但正常 teammate 流程不要求使用该开关。
- 在 admin token 创建成功后生成完整 teammate `.env`，并让 collector 支持 `CLI 参数 > ~/.token-meter/collector.env > 系统环境变量` 的配置优先级。
- 复用 P3 device token、team/user/device binding、team ingestion、team report 基础设施。
- Dashboard 的 Local 和 Team 视角都能按 tool 区分 Codex 和 Claude Code。
- 不采集 prompt、response、raw API body、raw transcript、用户输入或模型输出正文。

产出物：

- `docs/P4-2026-05-01-README.md`
- `docs/contracts/P4-2026-05-01-claude-code-usage-event.md`
- `docs/contracts/P4-2026-05-01-claude-code-ingestion-source.md`
- `docs/contracts/P4-2026-05-01-tool-usage-report-extension.md`
- `docs/milestones/P4-claude-code-team-collection/P4-2026-05-01-design.md`
- `docs/milestones/P4-claude-code-team-collection/P4-2026-05-01-tasks.md`
- `docs/acceptance/P4-2026-05-01-claude-code-team-collection.md`

验收标准：

- Codex 和 Claude Code 都能通过 teammate collector 上报到同一 team report。
- Local report 可展示 Codex 与 Claude Code，并支持 `tool=codex|claude-code` 筛选。
- Team report 可按 `tool`、`user`、`device`、`model`、`date` 聚合。
- Admin 页面能生成 teammate `.env`；collector 能读取客户端 env 文件，server 端不读取 teammate `.env`。
- Claude Code 上报不包含 prompt、response、raw API body 或 transcript 正文。
- P3 Codex team collection 行为不被破坏。
- Local/Team Day、Week、Month 对比继续可用。

## 阶段 5：Unified CLI Usage Metrics

目标：在 P4 已经跑通 Codex 和 Claude Code 统计的基础上，定义并验证一套统一 CLI usage metrics schema。Codex 和 Claude Code 必须映射到同一套 canonical usage event 字段，Local/Team report 必须按同一公式计算 summary、daily、models、sessions、tools 和 comparison。P5 不接入新 CLI，但 schema 必须能支撑未来 Cursor、Gemini CLI、自研脚本或本地网关接入。

P5 的重点不是抽象出完整 provider 平台，而是把“所有 CLI 工具最终都要映射成什么事实字段”和“report 层如何派生可比较指标”说清楚。

需要做的事：

- 定义 canonical usage event 事实字段：`tool`、`source_kind`、`source_quality`、`model`、`session_id`、`timestamp`、`input_tokens`、`output_tokens`、`cached_input_tokens`、`reasoning_output_tokens`、`total_tokens`，以及可选 `provider`。
- 明确派生指标只在 report 层计算：`net_input_tokens`、`net_total_tokens`、`cache_rate`。
- 统一 Codex 和 Claude Code 的字段映射、缺失字段 fallback 和无效 usage 过滤规则。
- 固化 P4 已实现的 tool filter、tool ranking、Local/Team tool 聚合和 period comparison 口径。
- 给 Codex 和 Claude Code 写独立 integration mapping 文档，覆盖本地路径、team collector `.env`、隐私边界、字段映射和排错流程。
- 检查并修正现有代码中 Codex/Claude Code 不一致的统计逻辑；只做让实现贴合 unified metrics contract 的最小代码调整。

不做：

- 不接 Cursor、Gemini CLI 或其他新 CLI。
- 不实现本地模型网关。
- 不实现 provider 自动路由。
- 不做费用估算、预算、告警或 billing。
- 不把 `net_*` 或 `cache_rate` 这类派生指标写入 DB，除非为了兼容既有字段必须保留。

产出物：

- `docs/P5-2026-05-24-README.md`
- `docs/contracts/P5-2026-05-24-unified-cli-usage-metrics.md`
- `docs/integrations/codex.md`
- `docs/integrations/claude-code.md`
- `docs/milestones/P5-unified-cli-usage-metrics/P5-2026-05-24-design.md`
- `docs/milestones/P5-unified-cli-usage-metrics/P5-2026-05-24-tasks.md`
- `docs/acceptance/P5-2026-05-24-unified-cli-usage-metrics.md`

验收标准：

- Codex 和 Claude Code 的 canonical usage event 字段含义一致。
- Local `/api/report` 和 Team `/api/team/report` 的同名 token 字段含义一致。
- `net_input_tokens`、`net_total_tokens` 和 `cache_rate` 在 Local/Team、summary/daily/models/sessions/tools/comparison 中按同一公式派生。
- usage schema 不依赖 Codex 或 Claude Code 私有字段；未来 CLI 只需新增 source mapping，不需要重新定义 dashboard 指标。
- prompt、response、raw JSONL、raw API body、transcript 正文、admin token、device token 明文和 token hash 仍不得进入 payload、DB、report、stdout 或日志。

## 阶段 6：OpenAI-compatible 本地网关 MVP

目标：让支持 OpenAI-compatible API 的工具通过本地网关调用模型，同时记录 usage。第一版只做最小可验收链路，不背 streaming、Responses API 或多 provider 自动路由。

需要做的事：

- 实现 `/v1/chat/completions` 非 streaming。
- 转发请求到一个 OpenAI-compatible upstream。
- 捕获响应 usage。
- 记录 latency、finish time 和失败请求。
- 不默认存储 prompt 正文。

产出物：

- `docs/P6-2026-05-01-README.md`
- `docs/contracts/P6-2026-05-01-openai-compatible-gateway.md`
- `docs/contracts/P6-2026-05-01-gateway-usage-event.md`
- `docs/milestones/P6-local-gateway/P6-2026-05-01-design.md`
- `docs/milestones/P6-local-gateway/P6-2026-05-01-tasks.md`
- `docs/acceptance/P6-2026-05-01-local-gateway.md`

验收标准：

- OpenAI SDK 可通过本地网关完成一次非 streaming 请求。
- 成功请求能统计 usage。
- 失败请求也能记录错误和耗时。
- dashboard 能展示 gateway 产生的数据。

## 阶段 7：多 provider 适配

目标：把 OpenAI、Anthropic、Gemini、DeepSeek、本地模型统一成一个 usage schema。

需要做的事：

- 设计 provider adapter 接口。
- 实现 OpenAI adapter。
- 实现 Anthropic adapter。
- 实现 Gemini adapter。
- 实现 OpenAI-compatible third-party adapter。
- 维护模型价格表，但价格表不阻塞 token 统计。

产出物：

- `docs/contracts/P7-2026-05-01-provider-adapter.md`
- `docs/contracts/P7-2026-05-01-pricing.md`
- `docs/acceptance/P7-2026-05-01-provider-openai.md`
- `docs/acceptance/P7-2026-05-01-provider-anthropic.md`
- `docs/acceptance/P7-2026-05-01-provider-gemini.md`

验收标准：

- 不同 provider 的 usage 能统一入库。
- provider 差异被 adapter 层隔离。
- dashboard 不直接依赖 provider 原始响应。

## 阶段 8：更多工具接入

目标：在 Codex 和 Claude Code 之外，继续接入 Cursor、命令行脚本和自研工具。

需要做的事：

- Cursor：验证其模型请求是否可配置到本地网关；如果不可行，研究日志型或系统代理型。
- 命令行脚本：提供最小 usage event 上报 CLI 或 SDK。
- 自研工具：提供 HTTP ingestion contract 和示例。
- 给每个工具写独立接入文档。

产出物：

- `docs/integrations/cursor.md`
- `docs/integrations/cli-scripts.md`
- `docs/integrations/custom-tools.md`

验收标准：

- 每个工具都有明确配置步骤。
- 每个工具都有至少一条真实统计样本。
- dashboard 能按 tool/provider/model 过滤。

## 阶段 9：产品化

目标：从开发工具变成本地可长期使用的软件。

需要做的事：

- 托盘应用或桌面壳。
- 自动启动后台服务。
- 配置页面。
- API key 本地加密存储。
- 数据导出。
- 数据备份。
- 升级迁移。

产出物：

- 安装包。
- 用户配置文档。
- 数据迁移文档。

验收标准：

- 普通开发者不需要读源码也能使用。
- 升级不丢历史统计。
- 本地隐私边界清楚。
