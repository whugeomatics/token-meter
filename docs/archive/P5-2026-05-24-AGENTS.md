## 1. Work Principles

- Follow docs before design, design before implementation, and implementation before acceptance.
- Do not write feature code without the phase README, contract, design, tasks, and acceptance baseline.
- Keep each phase scoped to its stated problem.
- Do not rely on chat context for durable project decisions; write important constraints into docs.
- Before touching auth, admin token, device token, or privacy-sensitive code, state the security impact.
- Do not store prompt text, response text, raw JSONL, user input, model output, admin token, device token plaintext, or device token hash in report surfaces or logs.
- Git commit messages use English conventional commits: `type(scope): description`.

## 2. P5 Scope

P5: Unified CLI Usage Metrics.

P5 unifies Codex and Claude Code usage into one canonical CLI usage metrics model. It covers canonical usage facts, fallback rules, Local/Team derived report formulas, dashboard metric naming, source mapping documents, and schema shape for future CLI tools.

P5 does not add a new CLI source. It does not implement Cursor, Gemini CLI, a local model gateway, provider auto-routing, pricing, budgets, alerts, login, billing, or cloud sync.


P5 derived report metrics:

```text
net_input_tokens = max(input_tokens - cached_input_tokens, 0)
net_total_tokens = net_input_tokens + output_tokens + reasoning_output_tokens
cache_rate = cached_input_tokens / input_tokens
```

`net_*` and `cache_rate` are report-layer metrics and must not be written as new historical facts.

## 4. P5 Review State

As of the P5 review on 2026-06-15, implementation and earlier verification records exist, but P5 should not be treated as fully closed until the review follow-ups are resolved.

Review follow-ups:

- Claude Code flat usage fallback currently double-counts cached input when `total_tokens` is missing.
- Team ingestion currently stores missing `total_tokens` as `0` instead of applying canonical fallback.
- The P5 privacy boundary says DB must not contain complete local paths, but local SQLite still stores `source_files.path`.
- Team ingestion accepts blank `source_kind` and `source_quality` though P5 marks them required.

Do not start P6 feature work until these P5 review findings are either fixed or explicitly resolved by contract changes.

## 5. Architecture Constraints

The project has three modules:

- `token-meter-core`: shared app/collector configuration, Codex session parsing, team payload DTOs, common DTOs, and utilities.
- `token-meter-app`: dashboard/server entrypoint, HTTP/API, admin, report, SQLite CRUD/store, SQL resources, and static resources. Only app may depend on SQLite for server storage.
- `token-meter-collector`: teammate-side lightweight reporting entrypoint, Claude Code usage source, collector `.env` reading, and periodic upload. It must not start an HTTP server or include dashboard/admin/static resources.

Shared app/collector behavior belongs in core only when both sides truly need it.

## 6. Privacy Boundary

P5 must not collect, upload, store in report payloads, or log:

- prompt text
- response text
- raw JSONL
- raw API body
- raw transcript
- user input
- model output
- tool input payload
- tool output payload
- admin token
- device token plaintext
- device token hash
- complete local paths, unless the P5 contract is explicitly narrowed to allow local DB path identity for incremental ingestion
