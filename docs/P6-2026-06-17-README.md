# P6 Incremental Team Collection

## Goal

P6 turns teammate collection from snapshot-style re-upload into true incremental upload.

The product goal is to make team token reporting dependable during continuous collection. A teammate collector should upload only events newer than the last successful upload cursor, while still retrying safely after failures.

P6 is not a gateway phase. It does not add a request proxy, local model gateway, provider routing, billing, pricing, budgets, cloud sync, or new sensitive data collection.

## Inputs

P6 must use data that already exists in the P5 model:

- canonical usage events for Codex and Claude Code;
- collector local state database;
- existing Team ingest idempotency by `event_key`;
- existing dimensions: team, user, device, tool, model, date.

## Current Problem

The collector already stores discovered events locally with `event_key` uniqueness, but upload currently reads all stored events in the configured lookback window and posts them again. The server deduplicates by `event_key`, so token totals stay correct, but upload health shows repeated duplicates and the process behaves like full-window re-upload instead of incremental upload.

Local dashboard ingestion is a separate path:

- Local Codex ingestion already uses `source_files` checkpoint plus `usage_events.event_key` idempotency.
- Local Claude Code ingestion is idempotent by `event_key`, but its source scan is not yet as checkpoint-driven as Codex.
- P6's primary fix is Team collector upload selection. If implementation touches Local ingestion, it should only tighten Claude Code local checkpoint behavior without changing Local report semantics.

## Scope

- Add a collector upload cursor stored in the collector state database.
- Cursor identity is at least `(last_successful_event_timestamp, last_successful_event_key)` so events sharing the same timestamp are not skipped.
- On each run, scan local logs as needed, using the cursor as the lower bound when it exists; use the configured initial lookback window only when no cursor exists.
- Upload only events after the cursor and up to the run cutoff time.
- Advance the cursor only after the upload succeeds.
- If upload fails, do not advance the cursor; the next run retries from the previous cursor.
- Keep an empty successful upload as a heartbeat when there are no new events, so Upload Health can show the device is alive without creating duplicate usage.
- Keep server-side `event_key` deduplication as defense in depth, not as the primary incremental mechanism.

## Non-Goals

- No local model gateway.
- No request forwarding proxy.
- No provider adapters or provider auto-routing.
- No pricing, cost estimation, budgets, billing, or alerting platform.
- No cloud account, cloud sync, or login system.
- No prompt, response, raw JSONL, raw API body, transcript text, user input, or model output collection.
- No dashboard BI expansion in P6.
- No complex failed-batch replay queue unless the minimal cursor strategy proves insufficient.

## Success Criteria

- Re-running the collector without new local events posts no usage events and does not create duplicate upload noise.
- New Codex and Claude Code events after the upload cursor are uploaded on the next run.
- Collector downtime longer than the default lookback window does not skip events after the last successful cursor.
- A failed upload does not advance the cursor; a later successful run retries the missed events.
- A successful no-new-events run still records upload health as a heartbeat.
- P1/P2 Local report compatibility, P3 Team report compatibility, and P5 token formulas remain intact.
